# Fixing glitch/flash/twitch in the EnergySurface fbm WebGL background (root causes + corrected GLSL/JS)

_Research report for the SameDayDesk frontend refresh (2026-06-20). Area: webgl-flicker._

## Key takeaways

- The time teleport is real and is the #1 bug. The rAF callback timestamp `t` is the same clock as performance.now() (MDN), so `uTime.value = t * 0.001` keeps advancing while the loop is paused by the IntersectionObserver/visibilitychange. On resume the shader jumps forward by the entire paused duration in one frame, causing a visible snap. Fix: accumulate your own delta-time clock and reset `lastT` on resume so paused time is never integrated.
- The brightness flash is caused by `uVel` feeding BOTH the time-scale AND the alpha (line 32 `t = uTime*(0.05 + uVel*0.30)` and line 39 `a = bands*mask*(0.16 + 0.30*uVel)`). Fast scroll spikes velocity, which simultaneously speeds the flow and brightens the layer, so you see a flash. Decouple: drive only time-scale (or only alpha) from velocity, clamp velocity, and smooth it with frame-rate-independent exponential damping.
- The velocity smoothing is frame-rate dependent. `u + (vel - u)*0.06` and `vel *= 0.92` (lines 113-115) are per-frame constants, so on 120Hz displays they decay twice as fast as on 60Hz (Rory Driscoll). Replace with `1 - exp(-lambda*dt)` damping driven by the accumulated dt.
- The scroll handler injects an instantaneous spike via `Math.max(vel, ...)` (line 62), which is a hold-the-peak, not a smooth signal. Switch to writing a raw target velocity and let the damped uniform chase it, so single fast scroll events cannot punch a flash through.
- The banding/flicker over `mix-blend-mode: screen` (CSS line 14) comes from quantizing a smooth gradient to 8-bit alpha, then amplifying it through screen blend over the radial glow. Fix in-shader with ordered dither: add Interleaved Gradient Noise of +/- 0.5/255 amplitude to the color/alpha right before output (frost.kiwi, Jimenez IGN).
- Guard the noise/hash against NaN at extreme coordinates. `hash()` uses `fract(p*p + ...)` with no domain bound; very large `t` (which grows unbounded with absolute time) inflates `p`, and float precision collapse in `fract` produces sparkle/twitch. Bounding total time and wrapping coordinates keeps the hash in a stable range.
- prefers-reduced-motion is only checked once at mount (line 48) and never re-checked. Add a matchMedia `change` listener that pauses the loop live (web.dev), and render one static frame instead of a blank canvas when motion is reduced.
- Keep `dpr` rounded and capped (already good at `Math.min(1.5, dpr)` line 84) but floor it to avoid fractional backing-store sizes that cause shimmer; OGL's default `preserveDrawingBuffer:false` is correct (do not enable it) and continuous rAF is fine here since the IO/visibility gating already throttles offscreen.

## Scope\n\nThis is grounded in the actual component at `/Users/plasmic/dev/web/samedaydesk/client/src/components/EnergySurface.tsx` and its stylesheet `/Users/plasmic/dev/web/samedaydesk/client/src/components/EnergySurface.module.css`. Every fix below references the specific lines that cause the artifact. The `prefersReducedMotion` helper lives at `/Users/plasmic/dev/web/samedaydesk/client/src/motion/gsap.ts:14`.\n\nThe four artifacts you describe all exist in this file. Here is each root cause, why it produces the visible glitch, and the corrected pattern.\n\n---\n\n## (1) The time \"teleport\" on resume (the snap/jump)\n\n### Root cause\n\nLine 112:\n```ts\nprogram.uniforms.uTime.value = t * 0.001;\n```\nThe `t` passed into a `requestAnimationFrame` callback is a `DOMHighResTimeStamp` measured from the same time origin as `performance.now()` (MDN: \"similar to calling `performance.now()` at the start of the callback\"). It is **absolute wall-clock time**, not elapsed animation time.\n\nWhen your `IntersectionObserver` or `visibilitychange` handler calls `pause()` (lines 120-125), `cancelAnimationFrame` stops the loop, but the clock `t` keeps running in the real world. When `play()` fires again, the very next frame sets `uTime` to a value that is larger by the **entire paused duration** (could be many seconds or minutes for a backgrounded tab). The fbm domain warp at lines 32-34 advances proportionally, so the smoke snaps to a completely different state in a single frame. That is your jump.\n\nThis is the classic rAF pause bug: MDN explicitly notes rAF \"calls are paused in most browsers when running in background tabs,\" so on `visibilitychange` you must reset your time reference on resume, never integrate the gap.\n\n### Fix: accumulate your own delta-time clock\n\nMaintain a private `elapsed` accumulator and a `lastFrame` marker. On every frame, add the clamped delta. On resume, reset `lastFrame` so the gap is discarded.\n\n```ts\nlet elapsed = 0;        // animation time, only advances while visible\nlet lastFrame = 0;      // timestamp of previous rendered frame\nlet running = false;\nlet raf = 0;\n\nconst loop = (t: number) => {\n  // First frame after (re)start: seed lastFrame, contribute zero dt.\n  if (lastFrame === 0) lastFrame = t;\n\n  // Clamp dt so a missed/slow frame or a resume slip cannot cause a jump.\n  // 0.05s = ~3 frames at 60Hz is a safe ceiling for a decorative bg.\n  const dt = Math.min(0.05, (t - lastFrame) * 0.001);\n  lastFrame = t;\n  elapsed += dt;\n\n  program.uniforms.uTime.value = elapsed;   // <- monotonic, gap-free\n\n  // ...velocity + render here (see section 2)...\n  raf = requestAnimationFrame(loop);\n};\n\nconst play = () => {\n  if (running) return;\n  running = true;\n  lastFrame = 0;            // <- force re-seed; discards the paused interval\n  raf = requestAnimationFrame(loop);\n};\nconst pause = () => {\n  if (!running) return;\n  running = false;\n  cancelAnimationFrame(raf);\n};\n```\n\nThe two load-bearing lines are `lastFrame = 0` in `play()` (drops the gap) and `Math.min(0.05, ...)` (caps any residual slip). MDN's own delta-time guidance is to derive progress from the timestamp argument rather than assume a fixed step, which also makes the animation speed consistent across 60Hz/120Hz displays.\n\nNote: because `elapsed` is now bounded by visible time only, it grows far more slowly than absolute `performance.now()`, which also helps section (4)'s precision issue.\n\n---\n\n## (2) The brightness flash from scroll velocity\n\n### Root cause\n\nThe velocity uniform `uVel` is wired into **two** places in the shader:\n\n- Line 32 (time-scale): `float t = uTime * (0.05 + uVel * 0.30);`\n- Line 39 (alpha/brightness): `float a = bands * mask * (0.16 + 0.30 * uVel);`\n\nSo a single fast scroll simultaneously (a) speeds up the flow and (b) ramps the layer alpha up by nearly 3x (0.16 to 0.46). Through the `screen` blend (CSS line 14) that alpha jump reads as a bright flash. Coupling brightness to velocity is the direct cause.\n\nMaking it worse, the smoothing is wrong on two counts:\n\n1. **The scroll handler holds the peak** (line 62): `vel = Math.min(1.4, Math.max(vel, (dy/dt)*7))`. `Math.max(vel, ...)` means a single fast scroll event slams `vel` to a high value instantly; it is a spike injector, not a smooth signal.\n2. **The decay is frame-rate dependent** (lines 113-115): `u + (vel-u)*0.06` and `vel *= 0.92` use fixed per-frame coefficients. On a 120Hz display those run twice per 60Hz-equivalent interval, so the layer decays/responds twice as fast, and the flash duration changes with refresh rate (Rory Driscoll: naive per-frame lerp \"smoothing will also be variable\").\n\n### Fix: decouple, clamp, and damp frame-rate-independently\n\n**Step 1: decouple brightness from velocity.** Pick one channel for velocity. For a calm background, keep alpha **constant** and let velocity only nudge the flow speed (a subtle, non-flashing effect). If you want a little life in the alpha, cap its velocity contribution to a tiny range (e.g. +/- 0.04) so it can never flash.\n\nShader, corrected:\n```glsl\n// Time-scale may use velocity (motion change is not a flash).\nfloat t = uTime * (0.05 + uVel * 0.18);\n\n// Alpha is essentially constant; velocity contributes at most ~0.04.\nfloat a = bands * mask * (0.18 + 0.04 * uVel);\n```\n\n**Step 2: write a raw target, never hold the peak.** The scroll handler should set a *target* the uniform chases, not jam the live value:\n```ts\nlet velTarget = 0;          // raw, set by scroll\nconst onScroll = () => {\n  const now = performance.now();\n  const dy = Math.abs(window.scrollY - lastY);\n  const dtMs = Math.max(16, now - lastT);\n  // Clamp the instantaneous reading to [0,1]; do NOT Math.max against the old value.\n  velTarget = Math.min(1.0, (dy / dtMs) * 6);\n  lastY = window.scrollY;\n  lastT = now;\n};\n```\n\n**Step 3: critically-damped / exponential smoothing driven by `dt`.** Replace the two magic per-frame constants with the frame-rate-independent form `1 - exp(-lambda*dt)` (Rory Driscoll; lisyarus). Inside `loop`, using the `dt` from section (1):\n```ts\n// velTarget also decays toward 0 so the effect settles when scrolling stops.\nvelTarget *= Math.exp(-3.0 * dt);          // target eases back to rest\n\nconst uVel = program.uniforms.uVel.value as number;\nconst lambda = 6.0;                        // higher = snappier, lower = smoother\nconst k = 1 - Math.exp(-lambda * dt);      // frame-rate-independent blend factor\nprogram.uniforms.uVel.value = uVel + (velTarget - uVel) * k;\n```\n\nThis is the exact `Damp(a, b, lambda, dt) = lerp(a, b, 1 - exp(-lambda*dt))` from Rory Driscoll, which gives identical perceived smoothing at 60Hz and 144Hz. Because the target itself is clamped to `[0,1]` and decays exponentially, and because alpha barely depends on it, fast scrolling can no longer punch a brightness spike through. lisyarus notes the same `lerp(current, target, 1 - exp(-speed*dt))` as the canonical \"exponential smoothing\" trick that tolerates large `dt` gracefully (relevant right after a resume).\n\nIf you want true ease-in as well as ease-out (no sudden initial motion), upgrade to a critically-damped spring (SmoothDamp / Game Programming Gems 4, Thomas Lowe): track a velocity-of-the-velocity term. For a decorative background, single-pole exponential damping is almost always enough and is cheaper.\n\n---\n\n## (3) Banding / flicker from `mix-blend-mode: screen` over the radial glow\n\n### Root cause\n\nCSS line 14: `mix-blend-mode: screen;` with `opacity: 0.6` over the `.glow` radial gradient (CSS lines 17-36). The shader outputs a smooth `bands = smoothstep(...)` ramp (line 35) into an 8-bit framebuffer. Quantizing a slow gradient to 256 levels produces visible Mach-band steps; `screen` blending then **lifts and stretches** those steps against the CSS glow, and any sub-LSB temporal wobble in the animated fbm makes adjacent frames cross quantization boundaries at different pixels, which reads as flicker. This is textbook gradient banding (shader-tutorial.dev; frost.kiwi).\n\n### Fix: ordered dither (Interleaved Gradient Noise) before output\n\nAdd a screen-space dither of one LSB amplitude, centered on zero so it does not shift overall brightness. The recommended function is Jorge Jimenez's Interleaved Gradient Noise (via frost.kiwi), which is cheaper and lower-pattern than a Bayer matrix and avoids white-noise sparkle:\n\n```glsl\n// Interleaved Gradient Noise (Jimenez). Screen-space, pattern-free.\nfloat ign(vec2 fragCoord){\n  return fract(52.9829189 * fract(dot(fragCoord, vec2(0.06711056, 0.00583715))));\n}\n```\n\nApply it to the final RGB and to alpha right before writing `gl_FragColor`, replacing line 40:\n```glsl\nfloat d = (ign(gl_FragCoord.xy) - 0.5) / 255.0;   // +/- 0.5 LSB, zero-mean\ncol += d;\na   = clamp(a + d, 0.0, 1.0);\ngl_FragColor = vec4(col, a);\n```\n\nThe `+/- 0.5/255` amplitude is exactly the frost.kiwi recommendation (`(1.0/255.0)*noise - (0.5/255.0)`). This pushes each pixel across the quantization threshold probabilistically, so the steps dissolve into imperceptible noise and the per-frame banding flicker disappears. For an even smoother result you can remap the noise to a triangular distribution (TPDF) by summing two IGN samples, which the Shadertoy banding-removal reference (loriendy/MslGR8) and Linelight write-up both advocate, but single-sample IGN at 1 LSB is usually enough for a decorative layer.\n\n**Also reduce the blend amplification at the source:** the combination of `opacity: 0.6` + `screen` is what magnifies residual steps. Consider lowering the in-shader alpha (section 2 already moves it toward a constant ~0.18) and dropping CSS `opacity` toward 1.0 so the dither, not the blend, controls the level. Keeping the canvas premultiplied-alpha-correct matters too: OGL creates the context with `premultipliedAlpha: true` by default, and you output straight alpha (`vec4(col, a)`), which is consistent here, but be aware that mismatches show up as edge halos under `screen`.\n\n---\n\n## (4) General hardening\n\n### NaN / sparkle in the hash at extreme coordinates\n\nLine 19:\n```glsl\nfloat hash(vec2 p){ p = fract(p * vec2(123.34, 456.21)); p += dot(p, p + 45.32); return fract(p.x * p.y); }\n```\nThis has no domain bound. In the original code `uTime` grew with absolute `performance.now()` and is multiplied into the fbm domain (lines 32-34), so over a long session `p` reaches large magnitudes. `highp float` is IEEE-754 single precision (~7 digits); once `p*123.34` exceeds the range where `fract` has sub-integer resolution, `fract` returns 0 or staircases, `p.x*p.y` can underflow/overflow, and you get sparkle, frozen tiles, or NaN that punches black/garbage pixels (a twitch). Two mitigations:\n\n1. Section (1) already bounds `elapsed` to visible time, dramatically slowing growth.\n2. Additionally wrap the time you feed the domain so coordinates stay small and the pattern loops seamlessly:\n```glsl\n// In JS, or in-shader: keep the animated offset in a bounded range.\nfloat tw = mod(uTime, 1000.0);   // wrap every ~1000s; fbm offsets stay finite\nfloat t = tw * (0.05 + uVel * 0.18);\n```\nKeep `highp` (you already declare it line 13), which on most mobile GPUs is required to avoid the worse `mediump` precision collapse. Inigo Quilez's domain-warp reference (iquilezles.org/articles/warp) is the canonical form your fbm follows; it gives no extreme-coordinate guidance, which is exactly why bounding time yourself is necessary.\n\n### Continuous rAF vs throttling\n\nYour continuous `requestAnimationFrame` (line 117) is correct here. The IntersectionObserver + visibilitychange gating (lines 122-125) already stops all work when the hero is offscreen or the tab is hidden, which is the meaningful battery/perf win. Do **not** add manual setTimeout throttling: it desyncs from vsync and reintroduces judder. MDN confirms the browser already pauses rAF in background tabs, so the visibility handler is belt-and-suspenders, plus it keeps your `elapsed` clock honest.\n\n### devicePixelRatio rounding\n\nLine 84 `dpr: Math.min(1.5, window.devicePixelRatio || 1)` is good (caps cost). For shimmer-free sampling, floor it so the backing store is an integer multiple and `gl.drawingBufferWidth/Height` (used for `uResolution`, line 103) are exact:\n```ts\ndpr: Math.min(1.5, Math.floor((window.devicePixelRatio || 1) * 2) / 2), // 0.5 steps\n```\nAlso re-read `uResolution` from `drawingBufferWidth/Height` after every resize (you already do, line 103), since fractional CSS sizes otherwise mismatch the aspect computation (line 30) and cause a one-frame stretch/twitch on resize.\n\n### preserveDrawingBuffer\n\nLeave it at OGL's default (`false`). You never read the canvas back and you re-render every frame, so `preserveDrawingBuffer: true` would only cost performance and can introduce its own stale-frame artifacts. Nothing to change; just do not enable it.\n\n### prefers-reduced-motion (currently only checked once)\n\nLine 48 checks `prefersReducedMotion()` a single time at mount and bails, leaving a blank canvas host. Two improvements per web.dev:\n\n1. **React live to changes** with a `change` listener, and pause/resume the loop instead of leaving a dead canvas:\n```ts\nconst mq = window.matchMedia(\"(prefers-reduced-motion: reduce)\");\nconst onMotionPref = () => (mq.matches ? pause() : play());\nmq.addEventListener(\"change\", onMotionPref);\ncleanups.push(() => mq.removeEventListener(\"change\", onMotionPref));\n```\n2. **Render one static frame** when motion is reduced rather than nothing: run `loop` once (or call `renderer.render` once with `elapsed` fixed) so users who prefer reduced motion still get the designed look, just frozen. web.dev's guidance is \"sometimes less movement is more,\" not \"no visual\": a single still frame respects the preference while keeping the brand surface.\n\n---\n\n## Suggested order of operations\n\n1. Fix the time teleport (section 1): biggest, most jarring artifact, and it also bounds the coordinate growth that feeds section 4.\n2. Decouple alpha from velocity and switch to `1 - exp(-lambda*dt)` damping (section 2): kills the flash.\n3. Add IGN dither + reduce screen-blend amplitude (section 3): kills the banding flicker.\n4. Add time-wrap + dpr floor + reduced-motion live handling (section 4): hardening.\n\nAll changes are confined to `EnergySurface.tsx` (the GLSL strings and the `loop`/`play`/`onScroll` handlers) and two CSS lines in `EnergySurface.module.css`. No token or design-system changes are needed.
